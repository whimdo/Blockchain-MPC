# 代币总览页接口设计文档

## 一、设计目标

当前代币总览页需要支持两类核心能力：

1. 页面进入时，一次性获取所有首页卡片的总览数据
2. 用户点击某张卡片的“更新”按钮后，仅刷新该卡片的数据并重新渲染

因此，页面当前阶段只需要两个接口：

- `GET /api/dashboard/tokens/overview`
- `POST /api/dashboard/tokens/refresh`

---

## 二、接口设计原则

### 1. 总览数据一次性加载
进入页面时，前端只发起一次请求，由后端统一返回全部卡片数据。

### 2. 单卡片局部刷新
点击“更新”按钮时，不重新请求整页数据，只更新当前卡片。

### 3. 刷新以 RPC 数据为主
单卡片更新时，后端应重新调用对应链的 RPC 服务获取实时状态数据，并结合价格服务或聚合逻辑返回该卡片最新状态。

### 4. 响应结构保持一致
单卡片刷新接口返回的卡片字段结构应尽量与总览接口中的单卡片结构一致，方便前端直接替换渲染。

---

# 三、接口一：代币总览接口

## 3.1 接口信息

### 接口名称
代币总览接口

### 请求方式
`GET`

### 请求路径
`/api/dashboard/tokens/overview`

---

## 3.2 接口用途

用于 Dashboard 首页首次加载时，一次性获取所有卡片所需数据。

该接口返回：

- 页面级时间信息
- 分组信息
- 每组下的卡片数据

---

## 3.3 请求参数

当前阶段建议：
**无必填参数**

可选参数后续如有需要再扩展，例如：

- `with_status=true`
- `symbols=BTC,ETH`

当前版本建议不加复杂参数。

---

## 3.4 返回结构设计

建议返回三层：

### 第一层：页面级字段
- `page_updated_at`
- `total_tokens`
- `group_count`

### 第二层：分组字段
- `group_name`
- `group_key`
- `cards`

### 第三层：卡片字段
- `symbol`
- `name`
- `display_name`
- `primary_chain`
- `category`
- `logo`
- `price`
- `price_display`
- `updated_at`
- `status`
- `tags`

---

## 3.5 返回字段说明

### 页面级字段

#### `page_updated_at`
类型：`string`  
说明：本次总览数据的统一生成时间

#### `total_tokens`
类型：`integer`  
说明：首页卡片数量

#### `group_count`
类型：`integer`  
说明：首页分组数量

---

### 分组字段

#### `group_name`
类型：`string`  
说明：分组名称，例如“核心资产”

#### `group_key`
类型：`string`  
说明：分组标识，例如 `core_assets`

#### `cards`
类型：`array`  
说明：当前分组下的卡片数组

---

### 卡片字段

#### `symbol`
类型：`string`  
说明：币种简称，唯一标识当前卡片

#### `name`
类型：`string`  
说明：币种全称

#### `display_name`
类型：`string`  
说明：前端展示名称

#### `primary_chain`
类型：`string`  
说明：该币种主链

#### `category`
类型：`string`  
说明：币种分类

#### `logo`
类型：`string`  
说明：卡片图标路径

#### `price`
类型：`number | null`  
说明：当前价格

#### `price_display`
类型：`string`  
说明：前端展示价格字符串

#### `updated_at`
类型：`string | null`  
说明：当前卡片对应数据更新时间

#### `status`
类型：`string`  
说明：当前卡片数据状态，例如：
- `online`
- `synced`
- `partial`
- `unavailable`

#### `tags`
类型：`array[string]`  
说明：标签信息

---

## 3.7 后端处理逻辑建议

该接口由后端统一聚合：

### 静态信息来源
来自：
- `dashboard_tokens_config.yaml`

包括：
- 名称
- symbol
- logo
- 主链
- 分类
- 标签

### 动态信息来源
来自：
- 价格服务
- RPC 状态服务
- 缓存或 MongoDB

包括：
- `price`
- `price_display`
- `updated_at`
- `status`

---

## 3.8 错误处理

### 成功
- HTTP 200

### 系统异常
- HTTP 500

返回建议：

```json
{
  "code": "DASHBOARD_OVERVIEW_ERROR",
  "message": "Failed to load dashboard token overview."
}
```

# 四、接口二：单卡片更新接口

## 4.1 接口信息

### 接口名称
单卡片刷新接口

### 请求方式
`POST`

### 请求路径
`/api/dashboard/tokens/refresh`

---

## 4.2 接口用途

用于用户点击某张卡片上的“更新”按钮时，仅刷新该卡片对应的数据。

该接口应触发后端重新执行以下动作：

1. 根据 `symbol` 找到对应币种配置
2. 根据该币种的主链调用对应 RPC 获取链状态或链上基础数据
3. 根据价格服务重新获取价格
4. 重新生成该卡片的动态展示字段
5. 返回该单张卡片的最新数据

---

## 4.3 请求参数

请求体建议使用 JSON。

### 请求字段

#### `symbol`
类型：`string`  
必填：是  
说明：需要刷新的币种标识

示例：

```json
{
  "symbol": "ETH"
}
```

---

## 4.4 响应结构设计

建议直接返回单张卡片对象，字段结构与总览接口中的单卡片保持一致，便于前端直接替换。

---

## 4.5 返回字段说明

#### `symbol`
类型：`string`

#### `name`
类型：`string`

#### `display_name`
类型：`string`

#### `primary_chain`
类型：`string`

#### `category`
类型：`string`

#### `logo`
类型：`string`

#### `price`
类型：`number | null`

#### `price_display`
类型：`string`

#### `updated_at`
类型：`string | null`

#### `status`
类型：`string`

#### `tags`
类型：`array[string]`

#### `refresh_source`
类型：`string`  
说明：此次刷新主要使用的数据来源，例如：
- `rpc`
- `rpc+price_service`
- `aggregated`

---

## 4.6 返回示例

```json
{
  "symbol": "ETH",
  "name": "Ethereum",
  "display_name": "Ethereum",
  "primary_chain": "ethereum",
  "category": "native",
  "logo": "/assets/tokens/eth.png",
  "price": 2055.12,
  "price_display": "2055.12 USDT",
  "updated_at": "2026-04-23T16:12:05Z",
  "status": "online",
  "tags": ["Layer1", "Native", "Smart Contract"],
  "refresh_source": "rpc+price_service"
}
```

---

## 4.7 后端处理逻辑建议

### 步骤 1：校验 symbol
确认该 `symbol` 是否存在于首页配置中。

### 步骤 2：读取静态配置
从 `dashboard_tokens_config.yaml` 中读取：
- 名称
- 主链
- 分类
- logo
- 标签

### 步骤 3：重新获取动态数据
建议至少刷新：

- 当前价格
- 当前状态
- 更新时间

如果你希望“点击更新按钮主要体现 RPC 的作用”，则这里的状态刷新应优先体现：

- RPC 是否在线
- 最近区块号是否可获取
- 链状态是否正常

### 步骤 4：重新构建单卡片结果
将静态信息与动态数据合并后返回。

---

## 4.8 错误处理

### 参数缺失
- HTTP 400

返回建议：

```json
{
  "code": "SYMBOL_REQUIRED",
  "message": "Field 'symbol' is required."
}
```

---

### 币种不存在
- HTTP 404

返回建议：

```json
{
  "code": "TOKEN_NOT_FOUND",
  "message": "Token configuration not found."
}
```

---

### RPC 获取失败
- HTTP 200 或 500 均可，建议根据系统策略选择

如果希望前端仍能保留卡片，只更新失败状态，建议返回 200，并附带失败状态：

```json
{
  "symbol": "ETH",
  "name": "Ethereum",
  "display_name": "Ethereum",
  "primary_chain": "ethereum",
  "category": "native",
  "logo": "/assets/tokens/eth.png",
  "price": null,
  "price_display": "--",
  "updated_at": "2026-04-23T16:12:05Z",
  "status": "unavailable",
  "tags": ["Layer1", "Native", "Smart Contract"],
  "refresh_source": "rpc"
}
```

# 五、前端调用建议

## 5.1 页面首次加载
进入页面时：

- 调用一次 `GET /api/dashboard/tokens/overview`
- 渲染全部卡片

---

## 5.2 点击单张卡片更新按钮
点击某张卡片右上角“更新”时：

- 仅调用 `POST /api/dashboard/tokens/refresh`
- 请求体传 `symbol`
- 用返回的单卡片数据直接替换当前卡片

---

## 5.3 不建议刷新全页
单卡片更新操作不建议重新调用 overview 接口，否则会导致：

- 页面刷新开销过大
- 用户体验不够聚焦
- 无法体现“局部更新”价值

---

# 六、最终设计总结

当前阶段代币总览页只需要两个接口：

## 1. 总览接口
- `GET /api/dashboard/tokens/overview`
- 用于首页一次性加载全部卡片

## 2. 单卡片刷新接口
- `POST /api/dashboard/tokens/refresh`
- 用于点击某张卡片更新按钮后，重新调用 RPC 和动态服务获取最新数据并替换当前卡片

---

# 七、一句话结论

> 当前代币总览页应采用“一个总览接口加载整页卡片 + 一个单卡片刷新接口局部更新”的设计方案，以兼顾页面加载效率、前端实现复杂度和后端数据聚合能力。
